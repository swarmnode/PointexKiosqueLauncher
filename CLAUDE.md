# CLAUDE.md

This file provides guidance to Claude Code (claude.ai/code) when working with code in this repository.

## Project overview

**PointexKioskLauncher** — an Android (Kotlin + Jetpack Compose) kiosk launcher app. It replaces the device home screen, restricts the device to a curated set of "Pointex"/"Fiducial" apps, and lets an administrator install new Pointex apps from an FTP update server. The app is designed to run as **Device Owner** so it can pin itself via lock-task mode and silently install APKs.

- Namespace / application ID: `com.pointex.kiosklauncher` (`.debug` suffix for debug builds)
- `compileSdk` 37, `minSdk` 26, `targetSdk` 35, Kotlin 2.3.21, AGP 9.2.1, Compose BOM 2026.05.00
- Published at `https://github.com/swarmnode/PointexKiosqueLauncher`

This is a multi-module Gradle build (`:app` is the kiosk launcher itself; `:qr-display-android` is a companion app, see below). The `qr-display-winforms/` directory is a separate, independent .NET project (not part of the Gradle build).

## Build commands

All commands run from the repo root (`C:\Users\filou\Launcher`), using the Gradle wrapper:

```powershell
# Build debug APK
.\gradlew.bat assembleDebug

# Build release APK (minified + shrunk resources)
.\gradlew.bat assembleRelease

# Install debug build on a connected device/emulator
.\gradlew.bat installDebug

# Lint
.\gradlew.bat lint
```

There are currently no unit or instrumented tests in `app/src/test` or `app/src/androidTest`.

### Device Owner provisioning

The kiosk restrictions in `KioskPolicyManager` only take effect once the app is provisioned as Device Owner (no other accounts on the device):

```powershell
adb shell dpm set-device-owner com.pointex.kiosklauncher.debug/.admin.KioskAdminReceiver
```

Without Device Owner, all `KioskPolicyManager` calls are safe no-ops (logged warnings), so the UI can still be exercised during development.

#### Field provisioning via QR code (no PC required)

Field technicians provision devices without adb using Android's QR-code Device Owner flow: factory-reset the device, tap 6 times on the setup wizard's welcome screen to open the QR scanner, and scan a QR code that points the device at the release APK. The device downloads, verifies (SHA-256 checksum) and installs the APK, then sets it as Device Owner automatically — the factory reset also clears any pre-existing Google account and Pointex app installs that would otherwise block this.

**Android 13+ limitation**: Google blocks `PROVISION_MANAGED_DEVICE` QR provisioning for DPCs not on its allowlist — confirmed via emulator (Android 13/Pixel_7) where replaying the exact QR intent fails immediately with "Can't set up device / Contact your IT admin", even though `adb install` + `adb shell dpm set-device-owner` on the same image works fine. Confirmed broken on a real Android 15 device. For Android 13+, use `tools/PrepaKiosquePointex-USB.bat` instead (USB + adb, see README "Android 13 et plus récent"). The script prefers an `adb.exe` (plus its 3 DLLs: `AdbWinApi.dll`, `AdbWinUsbApi.dll`, `libwinpthread-1.dll`) placed next to it — a portable technician kit, gitignored — and falls back to `adb` from the PATH.

`.github/workflows/release.yml` automates this for every tag push (`vX.Y.Z`): it builds the signed release APK, publishes a GitHub release with `PointexKiosqueLauncher.apk` (the Gradle output `app-release.apk` renamed; an identical asset is also uploaded under the old `app-release.apk` name so QR codes printed before the rename keep working — drop that transitional upload once none circulate), regenerates the QR with `tools/generate_provisioning_qr.py` pointing at the stable `releases/latest/download/PointexKiosqueLauncher.apk` URL, and uploads it as `provisioning_qr.png` on the same release. Because the download URL never changes, only the embedded checksum changes per release — no manual steps needed. The QR also embeds the workshop Wi-Fi credentials (`--wifi-ssid`/`--wifi-password`, from the `WIFI_SSID`/`WIFI_PASSWORD` repo secrets) so ManagedProvisioning has internet access to download the APK during OOBE — without this, devices with no saved Wi-Fi network fail provisioning with a generic "Can't set up device" error. The same workflow also builds and uploads the two QR-display companion apps below (`PrepaKiosquePointex.apk` from a `release`-job Gradle build, `PrepaKiosquePointex.exe` from a separate `windows-latest` job since `qr-display-winforms` needs `dotnet build` on Windows; the exe is uploaded bare — `GenerateSupportedRuntime=false` in the csproj means no `.exe.config` is produced, so it is a single file). The same Windows job finally assembles and uploads `PrepaKiosquePointex-Kit.zip`, the self-contained technician USB kit: `tools/PrepaKiosquePointex-USB.bat`, portable adb (4 files + NOTICE.txt extracted from Google's official `platform-tools-latest-windows.zip`), both APKs, the QR-display exe and the PDF notice generated by `tools/generate_kit_doc.py` (fpdf2, embeds Segoe UI from the runner). The kit pieces are also assembled locally in `tools/` (gitignored, see `.gitignore`) so the USB script works straight from a repo checkout.

To regenerate manually (requires `pip install "qrcode[pil]"`):

```powershell
python tools/generate_provisioning_qr.py app/build/outputs/apk/release/app-release.apk https://github.com/swarmnode/PointexKiosqueLauncher/releases/latest/download/PointexKiosqueLauncher.apk --wifi-ssid "..." --wifi-password "..."
```

#### QR-display companion apps

Two minimal, no-frills apps just display the current `releases/latest/download/provisioning_qr.png` image (nothing else) so an admin/technician can show the provisioning QR without opening a browser:

- **`qr-display-android/`** — Gradle module `com.pointex.provisioningqr`, single Compose screen that downloads and shows the QR. Build with `.\gradlew.bat :qr-display-android:assembleRelease`.
- **`qr-display-winforms/`** — .NET Framework 4.8 WinForms app (`PointexProvisioningQr`), a single form with a `PictureBox` showing the QR. Build with `dotnet build -c Release` from `qr-display-winforms/`.

## Architecture

### Single-activity, state-driven UI

`MainActivity` (`app/src/main/java/.../MainActivity.kt`) is the only activity (`launchMode="singleTask"`, registered as `HOME`/`LAUNCHER`). On every `onResume()` it calls `KioskPolicyManager.enterLockTask()`, which both pins the activity on first launch and re-locks the kiosk whenever the admin returns from Settings.

`KioskApp` (`ui/KioskApp.kt`) is the root composable and a small state machine over a private `KioskScreen` enum:

- `PROVISIONING_REQUIRED` → `ProvisioningRequiredScreen` — first-run screen shown when the app is neither Device Owner (`KioskPolicyManager.isDeviceOwner`) nor in chosen limited mode (`KioskModeRepository`); "Vérifier à nouveau" re-checks and proceeds to `HOME` once provisioned. A "Configurer en mode kiosque limité" option proceeds anyway in a degraded mode for devices that can never become Device Owner. In this mode `enterLockTask`/`exitLockTask` do **not** pin: screen pinning is Device Owner-only-silent — every re-pin otherwise triggers Android's confirmation dialog — and, since launching an allowed app requires being unpinned anyway (a pinned non-allowlisted task hits "Attempted Lock Task Mode violation", field-confirmed on Sunmi L2s PRO/Android 12), pinning would only ever cover the empty kiosk home screen and add no real protection. All `KioskPolicyManager` policies likewise stay no-ops. Limited mode instead rests on two mitigations: a "Définir un code de verrouillage" button opens `ACTION_SECURITY_SETTINGS` so the technician sets a device lock credential protecting system access (also set "Verrouiller automatiquement" to "Jamais" so the code is not asked on every screen wake), and choosing limited mode requests the default-launcher `ROLE_HOME` via `RoleManager` (Android 10+) so the Home button keeps returning to the kiosk. The choice is persisted (`KioskModeRepository`) because granting the HOME role recreates the activity. Limited-mode setup also activates `KioskAdminReceiver` as a plain device admin (`KioskPolicyManager.addAdminIntent` → `ACTION_ADD_DEVICE_ADMIN`, chained before the role request) so the app cannot be uninstalled through the normal paths; `onDisableRequested` warns before deactivation. This is not bulletproof without Device Owner — a user who knows to deactivate the admin in Settings → Security can still uninstall — but it blocks the casual/accidental paths. Device Owner devices already have the admin active (and could additionally use `setUninstallBlocked`, owner-only).
- `HOME` → `HomeScreen` — grid of allowed app tiles; a long-press in the bottom-right corner opens `AdminPinDialog`
- `FTP_INSTALL` → `FtpInstallScreen` — SFTP-based app installer

The allowed-apps list (`KioskAppRepository.getAllowedApps`) is refreshed on `ON_RESUME` and whenever it changes, `KioskPolicyManager.updateLockTaskPackages()` is called so those packages can be launched without breaking lock-task mode. Once configured (Device Owner, or limited mode chosen) and when exactly one allowed app is installed, `MainActivity.onResume` auto-launches it at device startup — once per boot (`BootLaunchRepository`, which detects reboots by comparing `SystemClock.elapsedRealtime()` against the stored value, immune to wall-clock/NTP changes), so deliberately returning to the kiosk home is never hijacked. On that pass, limited (non-Device-Owner) mode skips `enterLockTask` entirely: the screen-pinning confirmation dialog would otherwise appear on top of the auto-launched app.

Admin access is challenge-response, with no stored PIN and no setup step: `AdminPinDialog` displays a random 5-digit challenge and expects the nine's complement of its 3 middle digits (e.g. 81634 → 836, see `AdminCodeRepository`); a wrong answer regenerates the challenge. Solving it opens `AdminMenuDialog`, which lets the administrator open system Settings, Wi-Fi settings (e.g. to configure a static IP), SIM card settings, or manage Pointex apps (`FTP_INSTALL`). The three Settings options share `openSystemSettings()` in `KioskApp.kt`, which calls `KioskPolicyManager.exitLockTask()` (temporarily allowing `com.android.settings` in the lock-task allowlist) before launching the corresponding intent (`Settings.ACTION_SETTINGS`, `ACTION_WIFI_SETTINGS`, or `ACTION_NETWORK_OPERATOR_SETTINGS`); `enterLockTask()` on the next resume restores the kiosk lockdown. The SIM option first checks `hasSimCard()` (the operator-settings screen closes itself instantly with no SIM, which looks like a broken button) and shows a toast instead when no SIM is present.

### Admin/device-policy layer (`admin/`)

- `KioskAdminReceiver` — `DeviceAdminReceiver`; declared in the manifest with `android:permission="BIND_DEVICE_ADMIN"` and `res/xml/device_admin_policies.xml`.
- `KioskPolicyManager` — wraps all `DevicePolicyManager` calls: lock-task package allowlist, lock-task features (`LOCK_TASK_FEATURE_GLOBAL_ACTIONS or LOCK_TASK_FEATURE_SYSTEM_INFO` — blocks Home/Recents/notification shade but keeps the long-press-power Marche/Arrêt menu and the status-bar clock/battery/signal info), keyguard disable, and user restrictions (`DISALLOW_SAFE_BOOT`, `DISALLOW_FACTORY_RESET`, `DISALLOW_ADD_USER`, `DISALLOW_MOUNT_PHYSICAL_MEDIA`). It also registers the app as persistent preferred `HOME` activity (so the kiosk returns to the front after a reboot even if another launcher was the default) and `enterLockTask()` calls `ensureWifiEnabled()` to turn Wi-Fi back on if it was disabled (`setWifiEnabled` still works for Device Owners despite being a no-op for regular apps since API 29). Every method checks `isDeviceOwnerApp` first and is a no-op otherwise.

### Data layer (`data/`)

- `KioskAppRepository` — queries `PackageManager` for launchable activities and filters to package names containing `pointex` or `fiducial` (case-insensitive). Deliberately does **not** use `MATCH_DEFAULT_ONLY` since some Pointex launcher activities lack `CATEGORY_DEFAULT`.
- `PointexSftpRepository` — JSch (`com.github.mwiede:jsch`) SFTP client against a configurable host:22 (`/Versions-NF/Android/`). Lists per-app subfolders (picks the most recently modified `.apk`) or top-level `.apk` files, and downloads a selected app. Returns a `FtpResult<T>` (`Success`/`Failure`) with French, user-facing error messages. All calls are blocking — must run off the main thread (`Dispatchers.IO`). `StrictHostKeyChecking` is disabled (no bundled `known_hosts`), so the connection is encrypted but the server's host key isn't pinned. JSch's reflection-based class loading requires the `-keep`/`-dontwarn` rules for `com.jcraft.jsch.**` in `proguard-rules.pro`.
- `ApkInstaller` — streams a downloaded APK into a `PackageInstaller` session and commits it via a `BroadcastReceiver`/`PendingIntent`. When the app is Device Owner, both `install()` and `uninstall()` are silent (no confirmation dialog); on non-Device-Owner devices, both handle `STATUS_PENDING_USER_ACTION` by unpinning the kiosk (screen pinning blocks other apps' activities) and launching the system confirmation intent, then keep listening for the final result broadcast.
- `SecurePrefs` — internal helper providing cached `EncryptedSharedPreferences` (AES256-GCM master key via `MasterKey`/Android Keystore), shared file `pointex_kiosk_secure_prefs`.
- `AdminCodeRepository` — challenge-response admin code: random 5-digit challenge, expected response = nine's complement of the 3 middle digits. Verification is rate-limited (5 consecutive failures → 30 s lockout, persisted via `SecurePrefs` so it survives a reboot).
- `FtpCredentialsRepository` — saved SFTP connection details (`SftpCredentials`: host, username, password) via `SecurePrefs`, reused across `FtpInstallScreen` visits. Defines `DEFAULT_HOST` (`ftp.egcn.fr`) and `DEFAULT_USERNAME` (`tec`), used to pre-fill the connection form on first use.
- `KioskApp` (data class) — label, package name, activity name, and resolved `Drawable` icon for one allowed app.

### UI layer (`ui/`)

- `HomeScreen` — `FlowRow` grid of app tiles (icon + label); empty state offers "Installer une application Pointex".
- `FtpInstallScreen` — multi-step flow (`CREDENTIALS` → `LOADING` → `LIST` → `INSTALLING` → `RESULT`) for logging into the SFTP server, picking an app, downloading and installing it. The `CREDENTIALS` step has editable server address / username / password fields, pre-filled from `FtpCredentialsRepository` (or its `DEFAULT_HOST`/`DEFAULT_USERNAME` on first use); a successful connection saves all three. The `LIST` step also shows currently installed Pointex/Fiducial apps (`KioskAppRepository`) with a per-app "Désinstaller" button (confirmation dialog, silent uninstall via `ApkInstaller.uninstall`), so an admin can remove the old version before installing another.
- `AdminPinDialog` / `PinPad` — challenge display and 3-digit response entry for admin unlock (`PinDots`/`PinKeypad` keypad components).
- `AdminMenuDialog` — post-code menu offering "Gérer les applications Pointex", "Wi-Fi (adresse IP fixe)", "Carte SIM", "Ouvrir les Paramètres", a "Protection d'accès" toggle cycling the watchdog scope (désactivée → Paramètres seulement → toutes les apps), and "Activer le service de protection" (opens accessibility settings).
- `ChallengeCodeEntry` — shared challenge display + 3-digit response keypad used by both `AdminPinDialog` and `LockChallengeActivity`.
- `LockChallengeActivity` — full-screen gate the watchdog shows over a gated app; correct code reveals it (sets `WatchdogRepository.temporarilyUnlocked`), Back / "Retour au kiosque" returns to the kiosk.

### Access watchdog (`admin/KioskWatchdogService`)

Limited-mode-only feature (Device Owner lock-task already blocks everything): an `AccessibilityService` watching `TYPE_WINDOW_STATE_CHANGED`. When a gated foreground package appears it launches `LockChallengeActivity` to demand the admin code. Scope (`WatchdogRepository`, persisted via `SecurePrefs`): `DISABLED`, `SETTINGS_ONLY` (gates `com.android.settings`), or `ALL_NON_ALLOWED` (gates everything except the kiosk, the allowed apps in `WatchdogRepository.allowedPackages` fed from `KioskApp`, and an essential-system whitelist incl. SystemUI/IME/permission-controller/package-installer). Requires the user to enable the service once in Accessibility settings. Field-tested on Sunmi L2s PRO (Android 12): both scopes gate correctly, allowed apps pass, and the gate re-arms when the kiosk regains the foreground. Caveats: a brief flash of the gated app before the prompt covers it; quick-settings toggles in the notification shade are not gated (not foreground activities); OEM battery management may kill the service.
- `theme/` — Compose `Color`/`Theme` definitions (`PointexKioskLauncherTheme`).

### UI text language

All user-facing strings (including error messages from `PointexSftpRepository`/`ApkInstaller`) are in **French** — keep new UI copy consistent with this.
